# Convention Plugin: как создавать и использовать
Gradle Convention Plugins — когда общую конфигурацию сборки выносишь в переиспользуемые плагины, а не копируешь `build.gradle.kts` по всем модулям. Меньше дублирования, меньше боли, больше порядка. Почти как уборка, только полезнее.

## Что здесь происходит

В проекте есть отдельный included build:
- `my-build-src` — место, где живут convention plugins
- `some-service` — пример модуля, который применяет эти плагины

Идея простая:
1. В `settings.gradle.kts` подключается `includeBuild("my-build-src")`
2. Gradle видит плагины из `my-build-src` раньше, чем внешние источники
3. Модули проекта используют плагины по их ID, например:
   - `myco.spring-boot-app`
   - `myco.code-quality`

---

## Как устроен convention plugin

Convention plugin — это не “ещё один плагин ради плагина”, а способ:
- централизовать общие настройки
- задать единые правила для всех модулей
- уменьшить размер модульных `build.gradle.kts`
- облегчить поддержку и масштабирование проекта

### Типичная структура
- `settings.gradle.kts`  
  Подключает `my-build-src` как источник плагинов

- `my-build-src/build.gradle.kts`  
  Описывает зависимости, нужные самим convention plugins

- `my-build-src/src/main/kotlin/*.gradle.kts`  
  Здесь лежат precompiled script plugins — именно они и становятся convention plugins

- `some-service/build.gradle.kts`  
  Использует готовые plugin IDs и добавляет только модульную специфику

## Как создавать convention plugin

### 1. Подключить included build

В корневом `settings.gradle.kts` нужно включить `my-build-src`:

- через `pluginManagement { includeBuild("my-build-src") }`

Это позволяет Gradle находить ваши плагины локально, без публикации в plugin portal или maven repo.

### 2. Использовать `kotlin-dsl`

В `my-build-src/build.gradle.kts` обычно включают:

- `kotlin-dsl`

Это удобно для precompiled script plugins и даёт нормальный Kotlin DSL experience без лишней магии.

### 3. Добавлять зависимости плагинов в `implementation`

Если внутри convention plugin нужны API других плагинов, их зависимости добавляются в `implementation`, а не в блок `plugins {}`.

Почему так:
- `plugins {}` применяет плагин к текущему проекту
- `implementation` делает API доступным для кода convention plugin
- так проще писать настраиваемую логику, а не только декларативное подключение

### 4. Создавать plugin script файл

Имя файла определяет ID плагина.

Например:

- файл `myco.spring-boot-app.gradle.kts`
- plugin ID `myco.spring-boot-app`

Это удобно и прозрачно: по имени файла сразу понятно, что именно делает плагин.

## Как использовать convention plugin в модуле

В модуле достаточно подключить plugin ID:

- `id("myco.spring-boot-app")`
- `id("myco.code-quality")`

После этого модуль получает:

- общие настройки Java/Kotlin
- единые тестовые настройки
- Spring Boot конфигурацию
- quality tooling вроде Detekt и JaCoCo
- стандартизированное поведение сборки

При этом сам модуль остаётся чистым: в нём только уникальные зависимости и небольшая локальная конфигурация.

## Best practices

### 1. Держите модули тонкими
В модуле должно быть только то, что действительно отличается от остальных.  
Если один и тот же блок копируется в 3+ местах — почти наверняка ему место в convention plugin.

### 2. Делайте плагины узкими по смыслу
Лучше несколько небольших plugin’ов, чем один “комбайн на все случаи жизни”.

Хорошая практика:
- `code-quality`
- `spring-boot-app`
- `library-module`
- `publishing-conventions`

### 3. Не смешивайте инфраструктуру и доменную логику
Convention plugin должен отвечать за сборку, tooling и стандарты, а не за бизнес-логику приложения.

### 4. Используйте Version Catalog
Ссылки на зависимости и версии лучше хранить централизованно в `libs.versions.toml`, чтобы:

- не дублировать версии
- упростить обновления
- избежать “почему тут 3.3.5, а там 3.3.4?”

### 5. Не жестко зашивайте всё в один модуль
Если настройка нужна только части проекта, не делайте её глобальной.  
Convention plugin должен быть удобным, но не навязчивым.

### 6. Выносите общие quality checks в отдельный plugin
Отдельный плагин для качества кода — хорошая идея, потому что он может включать:

- Detekt
- Jacoco
- coverage verification
- общие правила CI-friendly отчётности

### 7. Настраивайте разумные дефолты
Convention plugin должен задавать:

- разумные значения по умолчанию
- единый стиль
- предсказуемую сборку

Но при необходимости оставлять возможность локальной донастройки.

### 8. Документируйте plugin IDs и назначение
Когда через полгода вы откроете проект и увидите `id("myco.spring-boot-app")`, должно быть очевидно, что именно этот плагин делает.

### 9. Проверяйте изменения через отдельный example module
Полезно иметь один модуль-пример, который использует плагины “как настоящий потребитель”.  
Это помогает быстро заметить, если convention plugin стал слишком тяжёлым или что-то сломал.

### 10. Следите за читаемостью
Convention plugin должен уменьшать шум в проекте, а не добавлять ещё один слой абстракции ради красоты.

## Практические советы по организации

### Хорошо, когда plugin делает:
- подключение стандартных плагинов
- настройку toolchain
- конфигурацию тестов
- настройку отчётов
- единые quality rules

### Плохо, когда plugin:
- скрывает слишком много поведения
- содержит сложные условные ветвления
- зависит от неочевидных побочных эффектов
- делает модульную сборку “магической”

---

## Когда convention plugin особенно полезен

- много модулей с одинаковой базовой конфигурацией
- нужен единый стандарт качества кода
- проект растёт, и копипаста в `build.gradle.kts` начинает раздражать
- хочется безопасно менять правила сборки в одном месте
- необходимо поддерживать consistent build logic для команды

---

## Короткий чеклист

Перед тем как выносить конфигурацию в convention plugin, спросите себя:

- это повторяется в нескольких модулях?
- это действительно общая логика сборки?
- это не слишком специфично для одного модуля?
- это улучшит поддержку проекта?
- это можно описать понятным и переиспользуемым образом?

Если на большинство вопросов ответ “да” — скорее всего, пора в plugin.

## Итог

Convention plugin — это способ сделать Gradle-сборку:

- чище
- предсказуемее

- удобнее для команды

Главная цель — не спрятать конфигурацию, а централизовать её там, где это действительно помогает.
